# ADR-002: Speaker diarization strategy

**Status:** accepted  
**Date:** 2026-07-22

## Context

Диаризация «кто говорит» критична для правильной атрибуции поручений. Текущий подход — `_silence_speakers`: новый speaker_id назначается при паузе >1.5 сек. Это даёт 4+ мнимых «спикера» для 3 реальных участников. SPEAKER_00, SPEAKER_02 — часто один и тот же человек. LLM на основе этих меток выдумывает имена (Ivan, Gena), и поручения назначаются не тем людям.

## Decision

**Трёхуровневая стратегия:**

1. **Базовый уровень (сейчас, всегда доступен):** silence-gap heuristic + `--participants SPEAKER_00=Николай,SPEAKER_01=Алексей` — пользователь вручную мапит метки на имена. LLM в промпте запрещено выдумывать имена.

2. **Продвинутый уровень (опционально):** `pip install meeting-intelligence[diarization]` → pyannote.audio 3.1. Настоящая диаризация по голосовым эмбеддингам: разные голоса → разные спикеры. Требует ~200MB модели, работает локально.

3. **Интерактивный уровень (в разработке):** агент после транскрибации показывает sample quotes каждого SPEAKER_NN через `clarify` и спрашивает «кто это?». Пользователь отвечает — агент переразмечает протокол.

## Consequences

- **+** Базовый уровень работает без дополнительных зависимостей
- **+** `--participants` флаг решает 80% проблемы атрибуции
- **+** pyannote опционален, не тянется в core deps
- **−** Silence-gap heuristic ошибается при быстрой смене говорящих
- **−** Интерактивный режим требует участия пользователя (не autonomous)
