# ADR-003: Protocol extraction — two-pass verification

**Status:** accepted  
**Date:** 2026-07-22

## Context

Первый прогон протокола через Qwen-2.5-7B стабильно пропускает ~40% решений и поручений, путает исполнителей и выдумывает участников. Проблема не в конкретной модели — маленькие LLM (7B) не способны за один проход охватить 500+ строк транскрипта и выдержать строгую схему JSON.

## Decision

**Two-pass extraction:**

1. **Pass 1 (extraction):** LLM извлекает сырой протокол (participants, decisions, assignments) с source_quotes. Использует SPEAKER_NN, не выдумывает имена. Температура 0.1.

2. **Pass 2 (verification):** тот же или более сильный LLM получает исходный транскрипт + протокол из Pass 1. Задача: «Проверь протокол. Найди: пропущенные решения, неверных исполнителей, выдуманных участников, off-topic который ошибочно стал поручением. Верни исправленный JSON.»

3. **Fuzzy validation (между pass 1 и 2):** 60% word overlap вместо exact match для source_quote. Транслитерация имён → warning, не error.

4. **Garbage filter:** строки из 5+ одинаковых односимвольных токенов (Whisper-мусор «с с с с») удаляются до отправки в LLM.

## Consequences

- **+** Качество протокола растёт без смены модели
- **+** Второй проход ловит hallucinations первого
- **+** `MEETING_PROTOCOL_VERIFY=false` позволяет отключить verify pass для скорости
- **−** Двойной вызов LLM = 2× стоимость токенов и времени
- **−** На слабых моделях verify pass может внести новые ошибки
